## 3. En session réelle (navigateur)

1. Créer une session avec une Feature contenant au moins une zone d'ombre (ex. seuil de taille non précisé) ; mener l'**étape 1 — Interview** (plan S2.6) jusqu'à ce que le registre contienne au moins une hypothèse `en_attente` (panneau « Hypothèses à valider »).
2. Répondre à la question d'interview qui recouvre cette hypothèse en la **tranchant explicitement** (ex. « le seuil est bien de 10 Mo, confirmé par la MOA »).
3. Attendu dans la réponse du moteur : une ligne `[LEVÉE PROPOSÉE : #<id> — confirmée — …]` (le registre en attente est entré au prompt) ; le panneau « Dernier échange » affiche « **n levée(s) d'hypothèse proposée(s)** — la décision vous revient… (A8) ».
4. Attendu dans le panneau « Hypothèses à valider » (après rechargement aussi — la proposition est persistée) : badge **« Levée proposée : Confirmer »** + justification en italique + rappel « c'est vous qui décidez (A8) », **boutons Confirmer/Rejeter toujours présents**.
5. **Contre-épreuve A8 (le cœur du test)** : recouper en base —

```bash
kubectl exec -it <pod-postgres> -- psql -U <user> -d <db> -c \
  "SELECT id, statut, statut_propose, justification_proposee FROM workflow_hypotheses WHERE session_id = <id>;"
```

Attendu : `statut = 'en_attente'` **inchangé** alors que `statut_propose` est renseigné — la proposition n'a rien levé.

6. Cliquer « Confirmer » sur l'hypothèse proposée → `statut = 'confirmee'`, `decidee_le` renseigné : la décision individuelle reste le seul chemin de levée.

## 4. Cas dégradés

1. Réponse du PO qui ne tranche rien → aucune ligne `[LEVÉE PROPOSÉE]`, aucun badge (silence = pas de proposition).
2. Si le modèle propose un numéro fantaisiste (`#99`) : rien n'est persisté, `levees_proposees` vide dans la réponse api (vérifiable par `curl POST /workflows/<id>/message`).
3. La validation d'étape « Oui » ne lève toujours rien (invariant S2.5 inchangé).
