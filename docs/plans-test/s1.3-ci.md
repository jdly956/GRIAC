# Plan de test S1.3 — CI minimale (GitHub Actions)

**Environnement cible** : GitHub (`jdly956/GRIAC`) — rien à exécuter sur le pod. Les TU/TNR locaux restent la règle avant push ; la CI rejoue exactement `make lint` + `make test` + build des deux images.

## 1. CA1 — le pipeline tourne et passe sur la PR

1. Ouvrir la PR de la story (elle contient `.github/workflows/ci.yml`) — le workflow se déclenche sur `pull_request`.
2. Onglet **Checks** de la PR : **3 jobs** attendus, tous verts —
   - `lint-tests` (ruff + pytest, workspace complet) ;
   - `build-images (api)` et `build-images (web)` (build des Dockerfiles, cible `runtime`, images non poussées).
3. Résultat observable : chaque check conclut `success` ; le récapitulatif de la PR affiche « All checks have passed ».

## 2. CA2 — échec du pipeline = merge bloqué (action référent, une fois)

Sur GitHub : **Settings → Branches → Add branch protection rule** (ou **Rules → Rulesets**) sur `main` :

1. « Require status checks to pass before merging » ;
2. sélectionner les checks `lint-tests`, `build-images (api)`, `build-images (web)` (ils n'apparaissent dans la liste qu'après leur premier run — donc après la première PR avec CI) ;
3. enregistrer, puis vérifier sur la PR : la zone de merge affiche « Merging is blocked » tant que les checks ne sont pas verts.

⚠️ **Limite connue** : sur un dépôt **privé** en plan GitHub **Free**, les protections de branche ne sont pas applicables (option grisée / bannière « upgrade »). Dans ce cas, le CA2 est porté par : (1) les checks rouges bien visibles sur la PR, (2) la règle CLAUDE.md « chaque PR est revue par le référent avant merge ». Consigner le constat (protection activée ou non) dans `SESSIONS.md`.

## 3. Optionnel — vérifier le blocage en conditions réelles

Ouvrir une PR jetable avec un échec volontaire (ex. un test qui `assert False`), constater le check rouge (et le blocage du merge si la protection est active), puis fermer la PR sans merger.

## 4. Clôture

Consigner dans `SESSIONS.md` : numéro de la PR, checks observés verts (CA1), état de la protection de branche (CA2), et cocher les CA du backlog en conséquence.
